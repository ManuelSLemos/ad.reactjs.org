Revisitemos este demo de la pgina anterior acerca de Suspense para la carga de datos.

Cuando hacemos clic en el botn Next para cambiar el perfil activo, los datos de la pgina existente desaparecen inmediatamente y vemos el indicador de carga para todo la pgina nuevamente. Cundo ocurre la actualizacin?

Esto no era muy difcil de implementar. Sin embargo, si empiezas a pensar sobre cmo es posible que esto funcione, puede crear algunos pequeos cortocircuitos. Si actualizamos el estado, cmo puede pasar que no veamos el resultado inmediatamente? Dnde se est renderizando el nuevo <ProfilePage`?

Claramente, ambas versiones de <ProfilePage> existen al mismo tiempo. Sabemos que la antigua existe, porque la vemos en la pantalla e incluso muestra un indicador de progreso en ella. Y sabemos que la nueva versin tambin existe en algn sitio, porque es la que estamos esperando!

Pero, cmo existen dos versiones del mismo componente al mismo tiempo?

Esto tiene que ver con la esencia del Modo Concurrente. Anteriormente hemos dicho es un poco como si React trabajara en la actualizacin del estado en una rama. Otra forma de conceptualizarlo es que al envolver la actualizacin del estado en startTransition comienza a renderizarlo en un universo diferente, como en las pelculas de ciencia ficcin. No vemos ese universo directamente, pero podemos obtener una seal desde l dicindonos que algo est pasando (isPending). Cuando la actualizacin est lista, nuestros universos se mezclan, y vemos el resultado en la pantalla!

Juega con poco ms con el demo, e intenta imaginar que esto ocurre.

Por supuesto, dos versiones del rbol de renderizado al mismo tiempo es una ilusin, justo como la idea de que todos los programas se ejecutan en tu computadora al mismo tiempo es una ilusin. Un sistema operativo cambia entre diferentes aplicaciones muy rpidamente. De manera similar, React puede cambiar entre la versin del rbol que ves en la pantalla y la versin que est preparando para mostrar luego.

Una API como useTransition te permite enfocarte en la experiencia de usuario deseada, y no pensar en los mecanismos de su implementacin. An as, puede ser una metfora til imaginar que las actualizaciones envueltas en startTransition ocurren en una rama o en un mundo diferente.

Los tres pasos

Hasta ahora hemos analizado todos los diferentes estados visuales por los que puede atravesar una actualizacin. En esta seccin les daremos nombres y hablaremos sobre la progresin entre ellos.


Justo al final, tenemos el estado Completado (Complete). Ah es a donde queremos llegar eventualmente. Representa el momento en que la prxima pantalla se ha renderizado completamente y no est cargando ms datos.

Pero antes de que nuestra pantalla pueda estar Completada, podemos necesitar cargar algunos datos o cdigo. Cuando estamos en la prxima pantalla, pero algunas partes de ella todava se estn cargando, tenemos un estado de Esqueleto (Skeleton).

Finalmente, hay dos formas principales que nos llevan al estado de Esqueleto. Ilustraremos la diferencia entre ellos con ejemplos concretos.

Predeterminado: Retirado Esqueleto Completado

Abre este ejemplo y haz clic en Open Profile. Vers varios estados visuales uno por uno.

  • Retirada: Por un segundo, vers el fallback <h1>Loading the app...</h1>.
  • Esqueleto: Vers el componente <ProfilePage> y dentro <h2>Loading posts...</h2>.
  • Completado: Vers el componente <ProfilePage> sin componentes de respaldo dentro. Todo fue cargado.

Cmo separamos el estado de Retirada (Receded) y el de Esqueleto? La diferencia entre ellos es que el estado de Retirada se siente como un paso hacia atrs para el usuario, mientras el estado Esqueleto se siente como un paso hacia adelante en nuestro progreso para mostrar ms contenido.

En este ejemplo, comenzamos nuestro viaje en el componente <HomePage>:

<Suspense fallback={...}>
  {/* pantalla anterior */}
  <HomePage />
</Suspense>

Despus del clic, React comenz a renderizar la prxima pantalla:

<Suspense fallback={...}>
  {/* prxima pantalla */}
  <ProfilePage>
    <ProfileDetails />
    <Suspense fallback={...}>
      <ProfileTimeline />
    </Suspense>
  </ProfilePage>
</Suspense>

Tanto <ProfileDetails> y <ProfileTimeline> necesitan datos para renderizarse, as que se suspenden:

<Suspense fallback={...}>
  {/* prxima pantalla */}
  <ProfilePage>
    <ProfileDetails /> {/* se suspende! */}    <Suspense fallback={<h2>Loading posts...</h2>}>
      <ProfileTimeline /> {/* se suspende! */}    </Suspense>
  </ProfilePage>
</Suspense>

Cuando un componente se suspende, React necesita mostrar el fallback ms cercano. Pero el fallback ms cercano a <ProfileDetails> est en el nivel superior:

<Suspense fallback={
  // Vemos ahora este fallback a causa de <ProfileDetails>  <h1>Loading the app...</h1>}>
  {/* prxima pantalla */}
  <ProfilePage>
    <ProfileDetails /> {/* se suspende! */}    <Suspense fallback={...}>
      <ProfileTimeline />
    </Suspense>
  </ProfilePage>
</Suspense>

Es por eso que cuando damos clic al botn, se siente como si diramos un paso atrs. La barrera <Suspense> que estaba mostrando previamente contenido til (<Homepage />) tuvo que retirarse para mostrar el fallback (<h1>Loading the app...</h1>). Llamamos a este estado Retirada.

Mientras cargamos ms datos, React volver a intentar renderizar, y <ProfileDetails> puede renderizar satisfactoriamente. Finalmente, estamos en el estado Esqueleto. Vemos la nueva pgina con las partes faltantes:

<Suspense fallback={...}>
  {/* prxima pantalla */}
  <ProfilePage>
    <ProfileDetails />
    <Suspense fallback={
      // Vemos este fallback a causa de <ProfileTimeline>      <h2>Loading posts...</h2>    }>
      <ProfileTimeline /> {/* se suspende! */}    </Suspense>
  </ProfilePage>

Eventualmente, tambin se cargarn, y tenemos el estado Completado.

Este escenario (Retirada Esqueleto Completado) es el predeterminado. Sin embargo, el estado de Retirada no es muy placentero, porque esconde informacin existente. Es por eso que React nos deja optar por una secuencia diferente (Pendiente Esqueleto Completado) con useTransition.

Preferido: Pendiente Esqueleto Completado

Cuando utilizamos useTransition, React nos dejar permanecer en la pantalla anterior y mostrar all un indicador de progreso. Lo llamamos un estado Pendiente. Se siente mucho mejor que el estado de Retirada, porque ninguno de nuestro contenido existente desaparece, y la pgina permanece interactiva.

Puedes comparar estos dos ejemplos para sentir la diferencia:

  • Predeterminado: Retirada Esqueleto Completado
  • Preferido: Pendiente Esqueleto Completado

La nica diferencia entre estos dos ejemplos es que el primero utiliza <button>s corrientes, pero el segundo utiliza nuestro componente <Button> personalizado con useTransition.

Tren de revelacin de Suspense

Cuando ya estamos en la prxima pantalla, a veces los datos que se necesitan para desbloquear diferentes barreras <Suspense> llegan en una rpida sucesin. Por ejemplo, dos respuestas diferentes pueden llegar despus de 1000 ms y 1050 ms, respectivamente. Si ya has esperado un segundo, esperar por 50 ms no va a ser perceptible. Es por eso que React revela las barreras <Suspense> con un horario, como un tren que arriba peridicamente. As se intercambia una pequea demora por la reduccin de los desechos de la maquetacin y del nmero de cambios visuales presentados al usuario.

Puedes ver un demo aqu. Las respuestas de publicaciones (posts) y hechos curiosos (fun facts) llegan con 100ms entre ellas. Pero React las une y revela de forma conjunta sus barreras Suspense.

Recapitulacin

Lo ms importante que hemos aprendido hasta ahora es que:

  • Por defecto, nuestra secuencia de carga es Retirada Esqueleto Completado.
  • El estado de Retirada no se percibe muy bien porque esconde contenido existente.
  • Con useTransition, podemos optar por mostrar en su lugar primero un estado Pendiente. Esto nos mantendr en la pantalla anterior mientras se prepara la pantalla siguiente.
  • Si no queremos que algn componente retrase la transicin, podemos envolverlo en su propia barrera <Suspense>.
  • En lugar de llamar a useTransition en cada componente, podemos incluirlo dentro de nuestro sistema de diseo.

Cuando diseas componentes de React, comnmente es mejor encontrar la representacin mnima del estado. Por ejemplo, en lugar de mantener nombre, apellido y nombreCompleto en el estado, a menudo es mejor mantener solo nombre y apellido, y luego calcular nombreCompleto en el renderizado. Esto nos permite evitar errores donde actualizamos un estado, pero olvidamos el otro.

Sin embargo, en Modo Concurrente hay casos en los que quiz quieras duplicar algunos datos en diferentes variables de estado. Considera esta pequea aplicacin de traduccin:

const initialResource = fetchTranslation(initialQuery);

function App() {

  const [resource, setResource] = useState(initialResource);

    const value = e.target.value;
    setQuery(value);
    setResource(fetchTranslation(value));
  }

  return (
    <>
      <input
        value={query}
        onChange={handleChange}
      />

        <Translation resource={resource} />

function Translation({ resource }) {
  return (
    <p>
      <b>{resource.read()}</b>
    </p>
  );
}

Nota cmo cuando escribes en la entrada de texto, el componente <Translation> se suspende, y vemos el fallback <p>Loading...</p> hasta que obtenemos resultados nuevos. Esto no es ideal. Sera mejor si pudiramos ver la traduccin anterior por un tiempo mientras estamos cargando la nueva.

De hecho, si abrimos la consola, veremos una advertencia:

Warning: App triggered a user-blocking update that suspended.

The fix is to split the update into multiple parts: a user-blocking update to provide immediate feedback, and another update that triggers the bulk of the changes.

Refer to the documentation for useTransition to learn how to implement this pattern.

Como mencionamos antes, si alguna actualizacin de estado causa que un componente se suspenda, la actualizacin del estado debera estar envuelta por una transicin. Agreguemos useTransition a nuestro componente:

function App() {
  const [query, setQuery] = useState(initialQuery);
  const [resource, setResource] = useState(initialResource);
  const [startTransition, isPending] = useTransition({    timeoutMs: 5000  });
  function handleChange(e) {
    const value = e.target.value;
    startTransition(() => {      setQuery(value);
      setResource(fetchTranslation(value));
    });  }

  // ...

}

Prubalo en CodeSandbox

Intenta escribir ahora en la entrada de texto. Algo anda mal! La entrada de texto se est actualizando muy lentamente.

Resolvimos el primer problema (la suspensin fuera de una transicin). Pero ahora a causa de la transicin, nuestro estado no se actualiza inmediatamente. Y no puede manejar una entrada de texto controlada!

La respuesta a este problema es separar el estado en dos partes una parte con alta prioridad que se actualiza inmediatamente y otra parte con baja prioridad que puede esperar por una transicin.

En nuestro ejemplo, ya tenemos dos variables de estado. La texto de la entrada est en query, y leemos la traduccin de resource. Queremos cambiar el estado query para que ocurra inmediatamente, pero los cambios a resource (o sea, cargar una nueva traduccin) deben disparar una transicin.

As la solucin correcta sera poner a setQuery (que no se suspende) fuera de la transicin, y a setResource (que se suspender) dentro de ella.

function handleChange(e) {
  const value = e.target.value;
  
  // Fuera de la transicin (urgente)  setQuery(value);
  startTransition(() => {
    // Dentro de la transicin (puede ser postergado)
    setResource(fetchTranslation(value));
  });
}

Prubalo en CodeSandbox

Con este cambio, funciona como se espera. Podemos escribir en la entrada de texto inmediatamente, y la traduccin luego se pone al da con lo que hemos escrito.

Postergar un valor

Por defecto, React siempre renderiza una interfaz consistente. Considera un cdigo como este:

<>
  <ProfileDetails user={user} />
  <ProfileTimeline user={user} />
</>

React garantiza que en cualquier momento que miremos a estos componentes en la pantalla, reflejarn los datos del mismo user (usuario). Si se pasa un user distinto debido a una actualizacin del estado, los veras cambiar al unsono. No podrs nunca grabar una pantalla y encontrar un fotograma en el que se mostraran valores de distintos users. (Si se te presenta un caso como este, reporta el error!)

Esto tiene sentido en la gran mayora de las situaciones. Una interfaz inconsistente genera confusin y puede darle la impresin equivocada a los usuarios. (Por ejemplo, sera terrible si el botn de envo de la mensajera y el selector de conversacin no estuvieran de acuerdo sobre qu hilo de conversacin est seleccionado actualmente)

Digamos que la carga de los detalles de usuarios es muy rpida y toma solo 300 milisegundos. Actualmente, estamos esperando todo un segundo, porque necesitamos tanto los detalles como las publicaciones para mostrar una pgina de perfil consistente. Pero, y si queremos mostrar los detalles ms rpidamente?

Si estamos dispuestos a sacrificar la consistencia, podemos pasar potencialmente datos viciados a los componentes que retrasan nuestra transicin. Eso es lo que nos permite hacer useDeferredValue():

function ProfilePage({ resource }) {
  const deferredResource = useDeferredValue(resource, {    timeoutMs: 1000  });  return (

        <ProfileTimeline
          resource={deferredResource}          isStale={deferredResource !== resource}        />
      </Suspense>
    </Suspense>
  );
}

function ProfileTimeline({ isStale, resource }) {
  const posts = resource.posts.read();
  return (
    <ul style={{ opacity: isStale ? 0.7 : 1 }}>      {posts.map(post => (
        <li key={post.id}>{post.text}</li>
      ))}
    </ul>
  );
}

La concesin que estamos haciendo aqu consiste en que <ProfileTimeline> ser inconsistente respecto a otros componentes y potencialmente mostrar un elemento ms antiguo. Haz clic en Next varias veces, y lo notars. Pero gracias a eso, fuimos capaces de acortar el tiempo de la transicin de 1000 ms a 300 ms.

Que sea una concesin apropiada o no depende de la situacin. Pero es una herramienta til, especialmente cuando el contenido no cambia de forma muy visible entre los elementos, y el usuario puede ni siquiera darse cuenta de que estn mirando a una versin viciada por un segundo.

Cul es la diferencia entre esto y usar debounce? Nuestro ejemplo tiene una demora fija artificial (3 ms por cada uno de los 80 elementos), por lo que siempre va a existir una demora, sin importar cun rpida sea nuestra computadora. Sin embargo, el valor de useDeferredValue solo va detrs si el renderizado se toma un tiempo. No hay una espera mnima impuesta por React. Con una carga de trabajo ms realista, puedes esperar que la espera se ajuste al dispositivo del usuario. En mquinas rpidas, la espera podra ser menor o no existir, en mquinas lentas, sera ms notable. En ambos casos, la aplicacin se mantendra con la capacidad de responder. Esa es la ventaja de este mecanismo sobre debounce o throttle, que siempre imponen una espera mnima y que no impiden que se bloquee el hilo mientras se renderiza.

An cuando hay una mejora en la respuesta, este ejemplo an no es atractivo, porque le falta al Modo Concurrente algunas optimizaciones cruciales para este caso de uso.
